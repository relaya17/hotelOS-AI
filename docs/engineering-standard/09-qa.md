## 9.1 מדיניות איכות

כישלון בדיקות ב־PR Gate = אין Merge.

## 9.2 פירמידת בדיקות

| סוג | מטרה | חובה |
|-----|------|------|
| Unit | Domain + pure logic | כן |
| Integration | DB, API, queues | כן לגבולות |
| E2E | מסעי ליבה | כן למסכים קריטיים |
| Accessibility | WCAG | כן ל־UI |
| Security | OWASP checks / SAST/DAST לפי שלב | כן |
| Visual Regression | Design System + מסכים יציבים | כן ל־`packages/ui` |
| Performance | תקציבי ביצועים | כן לנתיבים חמים |
| Load | קיבולת | לפני scale milestones |
| Chaos | עמידות | בסביבות non-prod ייעודיות |
| DR Drills | שחזור | תקופתי |

## 9.3 כיסוי מינימלי (הוחלט, PO 2026-07-18)

- **Packages עם לוגיקה עסקית** (domain/application, למשל `packages/database`, `packages/auth`, feature packages): **70% line coverage** מינימום — חוסם ב־PR Gate.
- **Apps** (`apps/*` — הרכבת UI/routes מעל packages): אין סף % קשיח; חובה **UI smoke tests** למסעים קריטיים + **typecheck ירוק** כתנאי חוסם. E2E למסעי הליבה (§9.4) משלים את הכיסוי בפועל.
- אסור להוריד כיסוי בנתיבים קריטיים (booking, billing, auth, AI approvals) מתחת לסף הקיים — גם אם הוא מעל 70%.

## 9.4 E2E — מסעים חובה

1. יצירת הזמנה (public)  
2. Check-in אורח  
3. תשלום בסיסי  
4. משימת Housekeeping  
5. אישור AI עם Human Approval  
6. החלפת מלון ב־Multi Hotel (admin)

## 9.5 Performance & Load

- Performance budget ל־web (LCP/INP/CLS) — כרך 10 CI
- Load tests ל־booking ו־auth לפני השקות גדולות
- AI endpoints נמדדים בנפרד (latency + cost)

## 9.6 Chaos & DR

- Chaos Engineering רק עם גבולות בטוחים
- Disaster Recovery Drills: restore מגיבוי + תרגול תקשורת
- תוצאות מתועדות ב־Runbooks
