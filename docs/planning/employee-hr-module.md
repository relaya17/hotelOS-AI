# תכנון — מודול עובדים: הרשמה עצמית, מבחני יכולת, תעודת יושר

**סטטוס:** יישום שלב א׳ בקוד (הזמנות + פרופיל + קוד עובד + מסך ציבורי ב־Admin). שלבים ב׳–ד׳ חלקיים/בהמשך.
**תאריך:** 2026-07-17 · עודכן 2026-07-19

## רקע ומטרה

שלוש יכולות חדשות למערכת, כולן סביב "תיק עובד" מרכזי:

1. **הרשמה עצמית לעובד חדש** — עובד שהתקבל מקבל קישור/קוד הזמנה, נכנס מהמחשב שלו, ממלא את הפרטים בעצמו, ומקבל **קוד עובד** קבוע שדרכו כל הפעילות שלו במערכת מתועדת ונגישה כהיסטוריה אחת.
2. **מבחני יכולת / מיפוי עובדים** — מנהלים יכולים להקצות לעובדים מבחנים מקובלים בתעשיית המלונאות (שירותיות, ידע מקצועי, שפה וכו'), לראות תוצאות, ולהשתמש בהם במיפוי/שיבוץ.
3. **תעודת יושר (דף נקי מהרשעות)** — העלאת קובץ סרוק של התעודה הרשמית (משטרה / הרשות המנפיקה), עם מעקב סטטוס (ממתין / אושר / נדחה / פג תוקף) ואישור ידני של HR — **ללא** אימות אוטומטי מול המשטרה.

זהו מסמך תכנון בלבד — לא מתחילים לכתוב קוד עד לאישור.

## עקרון מארגן: תיעוד גנרי למסמכי עובד

כדי לא לבנות טבלה נפרדת לכל סוג מסמך (ת.ז, חוזה, תעודת יושר, תעודת הכשרה...), יכולת 3 (ותמיכה עתידית בעוד מסמכים) נבנית כטבלת מסמכים גנרית אחת: `employee_documents`. תעודת היושר היא רק `docType` אחד מתוכה.

## יכולת 1 — הרשמה עצמית + קוד עובד + היסטוריה

### זרימה מוצעת

1. HR/מנהל יוצר **הזמנה** במערכת (Admin): שם זמני, אימייל, מלון/מחלקה, תפקיד. המערכת יוצרת `employee_invites` עם טוקן חד-פעמי ותאריך תפוגה, ושולחת קישור (מייל/וואטסאפ ידני בשלב ראשון — אין צורך באינטגרציית שליחה אוטומטית ב-MVP).
2. העובד פותח את הקישור **מהמחשב שלו** — עמוד הרשמה עצמאית, ציבורי (לא דורש התחברות), מאומת רק דרך הטוקן בקישור (בדומה לתבנית הקיימת של `/v1/public/*`).
3. העובד ממלא: שם מלא, ת.ז, טלפון, כתובת, איש קשר לחירום, שפת ממשק מועדפת, וקובע סיסמה משלו.
4. בהגשה: נוצר `user` (role: `employee`), נוצר/מתעדכן `employee_profiles`, ומוקצה **קוד עובד** ייחודי קריא (למשל `EMP-2026-00123`) — לא ה-UUID הפנימי, אלא שדה ייעודי לתצוגה, סריקה בכניסה למלון, וכו'.
5. ההזמנה מסומנת כ"נוצלה" (`consumedAt`), לא ניתנת לשימוש חוזר.

### מודל נתונים

**הרחבת `employee_profiles`:**
- `employee_code` (TEXT, UNIQUE per tenant) — קוד עובד קריא.
- `phone`, `national_id`, `address`, `emergency_contact_name`, `emergency_contact_phone` (כל השדות הסטנדרטיים ל"הכל כמקובל").
- `status` (`active` / `pending` / `suspended` / `terminated`).

**טבלה חדשה `employee_invites`:**
```
id, tenant_id, hotel_id, department_id, email, role_hint,
invite_token_hash, created_by_user_id, expires_at, consumed_at, created_at
```

### "היסטוריה מול המערכת" — מסך מאוחד

עמוד פרופיל עובד (Admin, ואולי Executive ברמת רשת) שמרכז מתוך טבלאות **קיימות** + חדשות, הכל מסונן לפי `employee_id`:
- נוכחות (`attendance_events` — קיים כבר).
- מסמכים וסטטוסים (`employee_documents` — חדש, כולל תעודת יושר).
- תוצאות מבחנים (`assessment_results` — חדש, יכולת 2).
- אירועי אודיט רלוונטיים (`audit_events` — קיים, מסונן לפי `actor_user_id`).
- חדרי בריפינג/הודעות רלוונטיות (אופציונלי, בהמשך).

זו תצוגה מרוכזת (view/query), לא טבלה חדשה — "ההיסטוריה" היא צירוף של מה שכבר קיים ומה שנוסיף.

### API (טיוטה)

| נתיב | תיאור |
|---|---|
| `POST /v1/hr/invites` | יצירת הזמנה (מנהל/HR בלבד) |
| `GET /v1/public/hr/invites/:token` | שליפת פרטי ההזמנה (לצורך מילוי טופס) |
| `POST /v1/public/hr/invites/:token/complete` | השלמת הרשמה עצמית → יצירת user + employee_profile + employee_code |
| `GET /v1/hr/employees/:employeeId/history` | ההיסטוריה המאוחדת |
| `GET /v1/hr/employees` | רשימת עובדים (סינון לפי מלון/מחלקה/סטטוס) |

## יכולת 2 — מבחני יכולת / מיפוי עובדים

מבחנים סטנדרטיים בתעשיית המלונאות: שירותיות ואירוח, ידע מקצועי לפי תפקיד (פקידות קבלה, ניקיון, F&B), התאמה לצוות/מנהל (סגנון עבודה), שפה. המנהל מקצה מבחן, העובד עונה (מהמחשב/טלפון), התוצאה נשמרת בתיק העובד.

### מודל נתונים

**`assessment_templates`** — קטלוג תבניות מבחן (ניתן לנהל בעברית/אנגלית כמו שאר המערכת):
```
id, tenant_id (null = תבנית גלובלית משותפת), title_he, title_en,
category (service|role_knowledge|management|language),
passing_score, questions_json, created_at
```

**`assessment_assignments`** — הקצאת מבחן לעובד:
```
id, tenant_id, employee_id, template_id, assigned_by_user_id,
status (assigned|in_progress|completed|expired), due_at, created_at
```

**`assessment_results`**:
```
id, assignment_id, score, passed, answers_json,
completed_at, evaluated_by_user_id, notes, created_at
```

### API (טיוטה)

| נתיב | תיאור |
|---|---|
| `GET /v1/hr/assessment-templates` | קטלוג מבחנים |
| `POST /v1/hr/employees/:employeeId/assessments` | הקצאת מבחן |
| `GET /v1/hr/assessments/:assignmentId` | פרטי הקצאה + שאלות (לעובד) |
| `POST /v1/hr/assessments/:assignmentId/submit` | הגשת תשובות → ציון + עדכון תיק עובד |
| `GET /v1/hr/employees/:employeeId/assessments` | היסטוריית מבחנים לעובד |

### הערה

תוכן המבחנים עצמו (השאלות, סף המעבר) — צריך תוכן אמיתי מהנהלת המלון/HR. בשלב הראשון אפשר לטעון תבניות דמו גנריות ולתת ל-HR לערוך/להוסיף מהממשק.

## יכולת 3 — תעודת יושר (דף נקי מהרשעות) + מסמכי עובד כלליים

לפי ההחלטה: **העלאת קובץ סרוק בלבד** (PDF/תמונה), ללא אימות אוטומטי מול משטרה/רשות — רק מעקב סטטוס ידני.

### מודל נתונים

**`employee_documents`** (גנרי, תעודת יושר היא ערך אחד בתוך `doc_type`):
```
id, tenant_id, employee_id,
doc_type (criminal_record_clearance|id_card|contract|certification|other),
storage_key,           -- קובץ (ראו הערת אחסון למטה)
issuing_authority,     -- לדוגמה "משטרת ישראל"
issued_at, expires_at, -- תעודת יושר בד"כ עם תוקף מוגבל
status (pending_review|approved|rejected|expired),
reviewed_by_user_id, reviewed_at, notes,
uploaded_at, created_at
```

### הרשאות ופרטיות — חשוב

תעודת יושר היא **מידע רגיש במיוחד** (מידע על עבר פלילי). המלצות:
- גישה לצפייה/הורדה של קבצי `criminal_record_clearance` — רק לתפקיד HR ייעודי, לא לכל "מנהל" (Role חדש או flag נפרד, לא סתם `admin`).
- רישום כל צפייה/הורדה ב-`audit_events` (יש כבר תשתית audit קיימת — משתמשים בה).
- מדיניות שימור/מחיקה (retention) — כדאי לקבוע מראש כמה זמן שומרים אחרי סיום העסקה, בהתאם לדין הישראלי (לא קובעים כאן, זו שאלה משפטית שדורשת ייעוץ — המערכת רק צריכה לתמוך במחיקה מתוזמנת).

### API (טיוטה)

| נתיב | תיאור |
|---|---|
| `POST /v1/hr/employees/:employeeId/documents` | העלאת מסמך (multipart, כמו הקלטות בריפינג היום) |
| `GET /v1/hr/employees/:employeeId/documents` | רשימת מסמכי עובד |
| `GET /v1/hr/documents/:documentId/file` | הורדת הקובץ (הרשאה מוגבלת + audit) |
| `POST /v1/hr/documents/:documentId/review` | אישור/דחייה ע"י HR |

## נושאים חוצי-מערכת

### אחסון קבצים
מסמכי עובדים (כמו הקלטות בריפינג היום) דורשים אחסון קבצים אמיתי. ב-`docs/deployment/vercel.md` כבר סימנו שהקלטות (`RECORDINGS_PATH`) לא ישרדו על Vercel כי אין דיסק קבוע — **אותה מגבלה בדיוק חלה על מסמכי עובדים**. כשמגיעים לשלב הבנייה, כדאי לפתור את שתי הבעיות יחד עם פתרון אחסון קבצים מנוהל (Vercel Blob / S3 וכו'), במקום לפתור פעמיים.

### תפקיד (role) חדש: `employee`
היום התפקידים הקיימים הם בעיקר `admin` / `executive`. צריך תפקיד `employee` חדש עם הרשאות מצומצמות (לראות/לערוך רק את התיק שלו, למלא מבחנים שהוקצו לו, להעלות מסמכים משלו) + תפקיד/flag ל-HR עם גישה לתעודות רגישות.

### i18n
טפסי ההרשמה העצמית והמבחנים צריכים לתמוך ב-10 השפות הקיימות (בפרט שפות עובדים שאינן עברית/אנגלית — תאילנדית, אמהרית/ אחרות בפועל).

## סדר עבודה מוצע (Phases)

1. **שלב א׳** — סכימת בסיס: `employee_invites`, הרחבת `employee_profiles` (`employee_code` ושדות פרופיל), זרימת הרשמה עצמית מקצה לקצה, מסך "תיק עובד" בסיסי (נוכחות בלבד, כי זה כבר קיים).
2. **שלב ב׳** — `employee_documents` + תעודת יושר: העלאה, סטטוסים, הרשאות HR מוגבלות.
3. **שלב ג׳** — `assessment_templates/assignments/results`: קטלוג מבחנים בסיסי + הקצאה + מענה + ציון.
4. **שלב ד׳** — פתרון אחסון קבצים מנוהל (משותף למסמכי עובד + הקלטות בריפינג), לפני פריסה לפרודקשן.

## החלטות PO (2026-07-18)

Product Owner אישרה את כל המסמך ("הכל אמור להיות שלם מצידי") וסגרה את כל השאלות הפתוחות שלמעלה. אין עוד שאלות פתוחות למודול זה — מותר לתכנן/לממש לפי ההחלטות הבאות:

| # | שאלה פתוחה (מקור) | החלטה |
|---|---------------------|--------|
| 1 | באיזו אפליקציה נפתח קישור ההרשמה העצמית? | **מסך ציבורי בתוך Admin** (לא משטח/אפליקציה נפרדת) — נתיב ציבורי לא מאומת מלבד הטוקן, בדיוק בתבנית `/v1/public/*` הקיימת. אין הצדקה למשטח נפרד רק לטופס הרשמה חד-פעמי; חוסך תחזוקת אפליקציה חדשה. |
| 2 | מי רואה תעודת יושר? | **HR ייעודי בלבד** (role/flag נפרד מ־`admin` הגנרי) — **לא** מנהל מלון ישיר כברירת מחדל. כל צפייה/הורדה נרשמת ב־`audit_events`. מנהל מלון יכול לראות רק סטטוס מרוכז ("תקין/ממתין/חסר") בלי גישה לקובץ עצמו. |
| 3 | תבניות מבחן — global או tenant? | **Global כברירת מחדל** (`assessment_templates.tenant_id = null`) — קטלוג משותף שכל הרשתות משתמשות בו; כל tenant יכול להוסיף/לערוך תבניות פרטיות משלו (`tenant_id` מוגדר) בלי לשנות את הגלובליות. HR יכול לערוך את תוכן התבניות מהממשק. |
| 4 | מדיניות שימור מסמכים רגישים לאחר סיום העסקה | **הוחלט (PO + הערת יעוץ משפטי מוצמדת, לא תחליף לו):** נשמר **hash/flag בלבד** (סטטוס + hash לאימות שלמות, ללא הקובץ הסרוק) למשך **2 שנים** לאחר סיום העסקה — ראה [כרך 12 §12.9.1](../engineering-standard/12-compliance.md). **הקובץ הסרוק המלא אינו מאוחסן ב־HotelOS** — ניהולו off-platform (תיק פיזי / מערכת HR ייעודית) בכפוף לייעוץ משפטי פרטני של הרשת. המערכת תומכת במחיקה מתוזמנת אוטומטית של ה־hash/flag בתום 2 השנים. |

מסכם: אין blocker PO שנותר למודול זה. שלבי היישום (§ "סדר עבודה מוצע") נשארים כמתוכנן.
